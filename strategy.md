# Content Strategy — Everything Content

## Brand

**Tagline:** We reverse engineer trending AI products and rebuild what actually matters.

**Editorial mantra:** Find the hook. Decode the product. Rebuild the core.

**Target audience:**
- Builders
- AI-curious professionals
- Aspiring technical creators
- Founders / operators / developers

**Core content test — every piece must satisfy:**
1. Strong external hook
2. Clear challenge
3. Visible progress
4. Satisfying comparison
5. Transferable lesson

## 1. Hook Scoring Mechanism

Use this before committing to any newsletter issue or video.
Score each candidate from 1 to 5 on each dimension.

### Fast Filter (3 yes/no questions)

Before full scoring, ask:
1. Would someone click this in under 3 seconds?
2. Can we rebuild the core in 1-2 focused sessions?
3. Will the audience learn something reusable?

If the answer is not "yes" to all three, skip.

### A. Attention Strength (weight: 2)
Question: Will a smart internet-native person care instantly?
- 1 = obscure, no existing signal
- 2 = mildly interesting niche launch
- 3 = decent momentum in one community
- 4 = clearly trending in a visible place
- 5 = breakout object with obvious curiosity

Examples of high scorers: #1 Product Hunt launch, GitHub repo exploding in stars, viral X demo, Show HN thread with heavy engagement.

### B. Rebuildability (weight: 2)
Question: Can we recreate the core experience fast enough for content?
- 1 = too complex, too infra-heavy
- 2 = possible but not practical for your format
- 3 = hard but can replicate a stripped-down version
- 4 = feasible with clear scoping
- 5 = ideal for a fast reverse-engineer/build challenge

### C. Teachability (weight: 2)
Question: Will rebuilding this naturally teach something valuable?
- 1 = cool, but little transferable learning
- 2 = mostly spectacle
- 3 = one good lesson
- 4 = several reusable lessons
- 5 = deeply instructive while still entertaining

### D. Visual Payoff (weight: 1.5)
Question: Will the audience see obvious progress and a satisfying end result?
- 1 = mostly backend or invisible logic
- 2 = difficult to show on screen
- 3 = somewhat demo-able
- 4 = visibly satisfying
- 5 = highly screen-friendly and thumbnail-friendly

### E. Reveal Potential (weight: 1.5)
Question: Is there likely to be a surprising insight?
Examples: "The wow factor was mostly UX." "The hard part wasn't the AI." "This looked magical but was mostly orchestration." "The original moat is not where people think."
- 1 = no real insight beyond "we built it"
- 3 = one decent reveal
- 5 = multiple non-obvious takeaways

### F. Originality Safety (weight: 1)
Question: Can you cover this without feeling like a lazy clone?
- 1 = feels derivative / uncomfortable
- 2 = hard to differentiate
- 3 = possible with careful framing
- 4 = clearly transformative
- 5 = easy to make original by focusing on mechanism, lessons, or improvement

### G. Audience Fit (weight: 1.5)
Question: Does this fit your audience's identity?
- 1 = interesting, but wrong audience
- 3 = somewhat relevant
- 5 = perfect overlap

### H. Format Elasticity (weight: 1)
Question: Can this become multiple content assets?
Can it become: newsletter issue, long-form video, short clip, tweet thread / LinkedIn post, repo / template, community challenge
- 1 = one-off
- 3 = two formats
- 5 = content flywheel material

### Final Hook Score Formula
Hook Score = Attention strength x 2 + Rebuildability x 2 + Teachability x 2 + Visual payoff x 1.5 + Reveal potential x 1.5 + Originality safety x 1 + Audience fit x 1.5 + Format elasticity x 1

**Max score = 62.5**

Decision rule:
- 50+ = must cover
- 42-49 = strong candidate
- 34-41 = cover only if timing is right
- below 34 = reject

## 2. Content Pillars

### Pillar 1: Viral Rebuild
Format: A trending launch/demo/repo becomes a rebuild challenge.
Core question: Can we recreate the core experience ourselves?
Works best for: Product Hunt launches, GitHub repos, X demos, Show HN products
Example titles:
- We Rebuilt Today's #1 Product Hunt Launch
- This AI Demo Went Viral. We Tried to Recreate It
- We Reverse Engineered a Trending GitHub AI Tool

### Pillar 2: Hype Audit
Format: Take a hyped product and test what's real versus packaging.
Core question: Is this actually impressive, or just smart presentation?
Example titles:
- Is This Viral AI Product Actually Hard to Build?
- We Tested the Hype Behind This Trending AI Tool

### Pillar 3: Feature Extraction
Format: Rebuild only the killer feature, not the whole product.
Core question: What is the smallest piece that made this product spread?
Example titles:
- We Rebuilt the One Feature That Made This App Blow Up
- The Real Magic of This AI Product Was Just One Workflow

### Pillar 4: Better Than the Original
Format: Rebuild the core idea, then improve one important thing.
Core question: What would we change if we built this ourselves?
Example titles:
- We Rebuilt This Viral AI Tool — But Made It Better
- This Product Was Brilliant, But We Fixed Its Biggest Flaw

### Pillar 5: Build Under Constraint
Format: Add a rule to make it a game.
Constraint ideas: only Claude Code, only open-source tools, 90-minute limit, under $20, no frontend libraries, one-shot prompt only
Example titles:
- We Had 90 Minutes to Rebuild This AI Product
- Could Claude Code Recreate This Viral Tool in One Session?

### Pillar 6: Open-Source Equivalent
Format: Take a polished commercial idea and build the lean open version.
Core question: What's the simplest useful version anyone can build?
Example titles:
- We Built an Open Version of This Popular AI Product
- Here's the Lean Version of a Viral AI Startup

### Pillar 7: Lessons From the Build
Format: Meta-analysis from multiple rebuilds.
Core question: What patterns keep showing up in winning AI products?
Example topics:
- What 10 Viral AI Products Had in Common
- The UX Patterns Behind the Best AI Demos
- What Rebuilding 12 AI Products Taught Us

### Recommended Pillar Mix (per 12 issues)
- 5 Viral Rebuilds
- 2 Hype Audits
- 2 Feature Extractions
- 2 Better Than the Original
- 1 Lessons From the Build

## 3. Virality Scoring Mechanism

This is different from hook scoring. A hook can be good for building. A piece can still fail as content. Virality scoring tells you whether the story packaging is strong enough.

### A. Instant Curiosity (weight: 2)
Would the title alone trigger "Wait, what?"

### B. Stakes (weight: 1.5)
Is there a clear consequence or tension?
Examples: time limit, quality target, blind comparison, must match a feature, must launch same day, constrained toolset.
Low virality often comes from low stakes.

### C. Novelty (weight: 2)
Does it feel fresh, not like generic AI content?
Low-novelty: "Claude Code tutorial", "How to use AI agents", "Build an app with AI"
High-novelty: "We rebuilt a viral product", "We tested whether the hype was fake", "We isolated the feature that made this blow up"

### D. Visual Promise (weight: 1.5)
Can the thumbnail / hero image tell a story fast?
Good visual ingredients: original product vs your version, countdown timer, progress board, before/after, dramatic feature comparison

### E. Emotional Engine (weight: 1.5)
What emotion drives the piece?
Best emotions: curiosity, challenge tension, surprise, competence aspiration, skepticism, delight
If purely informative, virality drops.

### F. Shareability (weight: 1.5)
Would someone send this to a founder, builder friend, colleague, or AI enthusiast?
High-share reasons: "You should see how simple this actually was", "This breakdown explains why this product spread", "This is how you'd build it"

### G. Identity Pull (weight: 1.5)
Does consuming this make the viewer feel like: an insider, a smart builder, an early adopter, someone who understands what's actually happening?

### H. Clipability (weight: 1)
Can you extract 3-5 short moments?
Examples: "We found the real trick", "The hardest part had nothing to do with AI", "This took 4 prompts to crack"

### Final Virality Score Formula
Virality Score = Instant curiosity x 2 + Stakes x 1.5 + Novelty x 2 + Visual promise x 1.5 + Emotional engine x 1.5 + Shareability x 1.5 + Identity pull x 1.5 + Clipability x 1

**Max score = 57.5**

Decision rule:
- 46+ = high viral potential
- 38-45 = strong
- 30-37 = decent, but improve packaging
- below 30 = weak packaging even if the build is good

### Repackaging Rule
If Hook Score is high but Virality Score is low, do not drop the idea. Instead, repackage:
- Add a time constraint
- Narrow to one killer feature
- Add a blind test
- Frame it as myth-busting
- Compare your version against the original

## 4. Newsletter Template

The newsletter should feel like a premium field report, not a blog post. Fast to scan, story-driven, visual, easy to repurpose into video.

### Subject Line Patterns
- We Rebuilt Today's #1 Product Hunt Launch
- This AI Demo Went Viral. Here's How We'd Build It
- We Tried to Recreate a Trending GitHub Project
- The Real Trick Behind This Viral AI Product
- Can Claude Code Rebuild This?

Goal: Make the subject line sound like a mission, not a newsletter.

### Section 1: Opening Hook (3-5 lines max)
Template: what happened, why it got attention, what challenge you're taking on.

### Section 2: The Object of the Week
Compact profile card: name, source of discovery, why it's trending, one-line value proposition, public signal snapshot.
Visuals: hero screenshot, source badge, signal snapshot card.

### Section 3: Why It Worked
Three mini-subsections: What users saw, What made it feel magical, What probably drove the virality.
Visuals: annotated screenshot with arrows/callouts, magic breakdown diagram.

### Section 4: The Rebuild Brief
Include: objective, constraints, tool stack, what counts as success, what you're deliberately ignoring.
Visuals: challenge card, timer graphic, feature checklist, scope line graphic.

### Section 5: How We Think It Works
Break into: probable inputs, model/workflow, orchestration, UX layer, hidden complexity.
Visuals: simple architecture diagram, flowchart, block diagram.

### Section 6: The Build Log
Progression format: Attempt 1, what broke, what changed, Attempt 2, what finally worked.
Visuals: iteration screenshots, before/after, code snippets (only if visually useful), debugging snapshots, decision cards.
Important: Don't add too much raw code. Show code only when it teaches a specific decision.

### Section 7: Final Comparison
Compare: original product, your rebuilt version, what matched, what didn't, harder/easier than expected.
Visuals: side-by-side comparison, feature matrix, scorecard.

### Section 8: The Real Lesson
Format: Lesson 1: the wow factor was actually ___, Lesson 2: the hardest part was ___, Lesson 3: if you were building this from scratch, start with ___
Visuals: distilled framework card, reusable checklist.

### Section 9: Build Takeaway / Reusable Asset
Options: prompt template, mini architecture, starter repo, reusable workflow, design pattern, agent loop pattern, evaluation checklist.
Visuals: downloadable template preview, repository preview card, one-page cheat sheet.

### Section 10: Next Week's Tease
Keep an open loop. Examples: Next week we're testing whether a viral X demo is mostly UX theater. We found a GitHub repo exploding right now. We may try a blind comparison next.

### Required Visual System (every issue)
Must-have:
1. Hero screenshot (original product/demo/repo)
2. Signal card (upvotes, stars, engagement)
3. Annotated teardown screenshot (arrows, labels)
4. Build brief card (goal, constraints, success criteria)
5. Architecture diagram (simple inferred system flow)
6. Iteration gallery (2-4 progress screenshots)
7. Final side-by-side comparison (original vs rebuilt)
8. Lessons card (concise takeaways)

Nice-to-have: timeline graphic, timer/countdown, failure log card, harder/easier split card, blind test result card, benchmark chart, prompt/workflow card.

Rules: Every visual must prove it, explain it, or compare it. No huge code dumps, stock images, decorative visuals, or unannotated screenshots.
